# WordlyID + Cross-App References — Design

**Status:** draft
**Date:** 2026-05-15
**Scope:** New shared package (`~/github/wordly-id/`); backend additions in `~/chat/tasks/` and `~/github/write/sync/`; client additions in `~/github/do/` (DoKit + apps) and `~/github/write/` (WriteKit + apps). No changes to `sleeper-chat` core, articles, memory, or any other Sleeper service.

## Goal

Two slash commands, `/do` and `/write`, available inside both the Write editor and the Do task notes editor. As the user types `/do <query>` or `/write <query>` at a word boundary, a palette filters tasks or write notes by **title**. Selecting a result inserts a stable reference token at the cursor. The editor renders that token live as a chip showing the current title (and current status for tasks). Tapping a chip opens the referenced item — in-app when the reference is to the same kind, via URL scheme when it is to the other app.

The design is built to extend: future apps (e.g. a Links app) plug in as another *reference kind* without code edits to Write or Do beyond registering the new kind.

## Non-goals (v1)

- **Inline create.** Pressing Enter on `/do something not yet a task` does not create a new task. The palette only inserts references to items that already exist.
- **Reverse-link / backlink panel.** No "what links to this item" surface. The reference graph exists in storage; surfacing it is a v2.
- **Web rendering.** The chips render in the Write and Do native editors only. Markdown viewed elsewhere (GitHub, plain markdown previews) sees the raw `<do:DO-…>` / `<write:W-…>` autolinks.
- **Block-level transclusion.** A reference renders as a one-line chip — not an embedded excerpt of the target.
- **Server-side ID generation.** All `wordly_id`s are generated by the client and pushed up. Backends only validate format.

## 1. WordlyID package

New repo at `~/github/wordly-id/`. SwiftPM-only in v1 (no JS twin — see §6.2 for why).

### 1.1 Format

```
<PREFIX>-<WORD>-<WORD>-<WORD>
```

- Uppercase ASCII.
- Prefix is 1–4 letters (`W`, `DO`, `L`, …).
- Three words drawn from an embedded curated wordlist of ~2,500 words (concrete nouns and gentle adjectives, 3–9 letters, no slang, no brands, no offensive-list words, no homophones — leans Almanac-warm). Draws are partitioned by part-of-speech — see §1.3.
- Examples: `W-COPPER-DRIFTING-LANTERN`, `DO-RABBIT-WHISPERING-MAUVE`.

### 1.2 Public API

Two products in one repo:

- **`WordlyID`** — pure ID generation. No SwiftUI, no Foundation.URL, no networking.
- **`WordlyRefs`** — reference primitives (tokenizer, mirror, palette controller, chip view). Depends on `WordlyID` and SwiftUI.

```swift
// WordlyID
public enum WordlyID {
    public static func generate(prefix: String) -> String
    public static func generate(prefix: String, isUnique: (String) -> Bool) -> String
    public static func parse(_ id: String) -> (prefix: String, words: [String])?
    public static func validate(_ id: String) -> Bool
}
```

Collision policy: `generate(prefix:isUnique:)` retries up to **3 times** against the `isUnique` callback (which the caller wires to a local index lookup — see §4.2). On the fourth attempt it appends `-2`, `-3`, … until unique. The non-`isUnique` variant skips the check (used for tests and contexts where the caller asserts uniqueness elsewhere).

Math for the curious: with ~2,500 words and three-of-three draws, the space is 2,500³ ≈ 1.6 × 10¹⁰. Birthday-bound collision probability at N items is ≈ N² / (2 · W³). At N = 1,000 items: ~3 × 10⁻⁵. At N = 10,000: ~3 × 10⁻³. Personal-scale corpora are in the hundreds, so the realistic probability is in the 10⁻⁶ range. The retry exists to make the uniqueness contract honest, not because we expect to hit it often.

### 1.3 Wordlist

`Sources/WordlyID/Wordlist.txt`, one word per line, lowercased, sorted. Embedded as a `String` resource (Swift Package `resources: [.process("Wordlist.txt")]`). Selection: split the wordlist into roughly equal noun / adjective / verb-able partitions and draw one from each so IDs read with a faint sentence-like cadence (`COPPER-DRIFTING-LANTERN`). The partitioning is internal to the generator; the output is just three uppercased words separated by `-`.

The wordlist file itself is the v1 deliverable that needs human curation. The implementation plan will call out wordlist sourcing (likely: start from EFF Long Wordlist, filter by length + against an exclusion list, hand-bucket into the three partitions, target ~800 words per partition).

## 2. ReferenceKind protocol

A kind is what makes a reference "a write doc" vs "a task" vs (later) "a link". Each app implements `ReferenceKind`:

```swift
// WordlyRefs
public protocol ReferenceKind {
    static var prefix: String { get }           // "W", "DO"
    static var slashTrigger: String { get }     // "write", "do"  (typed after `/`)
    static var urlScheme: String { get }        // "write", "do"
    static var titlesEndpoint: URL { get }      // GET <base>/titles?since=<seq>
    associatedtype Item: ReferenceItem
}

public protocol ReferenceItem: Sendable, Codable {
    var wordlyID: String { get }
    var title: String { get }
    var mtime: Int64 { get }       // ms since epoch
    var deleted: Bool { get }
    var statusGlyph: String? { get }  // nil for kinds without a status concept
}
```

Write provides `WriteKind` (prefix `W`, scheme `write`) with `Item = WriteTitle`. Do provides `DoKind` (prefix `DO`, scheme `do`) with `Item = TaskTitle` and a non-nil `statusGlyph`. A future Links app provides `LinkKind` (prefix `L`, scheme `link`) — no edits to Write or Do.

Apps register the set of kinds they understand at startup:

```swift
WordlyRefs.registerKinds([WriteKind.self, DoKind.self])
```

The registered list drives the slash palette's available triggers, the tokenizer's prefix recognition, the chip renderer's lookup, and the URL-scheme dispatch.

## 3. On-disk syntax

```
<do:DO-RABBIT-DANCING-MAUVE>
<write:W-COPPER-DRIFTING-LANTERN>
```

Raw markdown autolink with a custom scheme. Properties:

- Valid CommonMark. Round-trips through any markdown parser as an autolink.
- ID-only — no embedded title — so the rendered chip is always the live title.
- Recognizable by a cheap regex: `<(scheme):(PREFIX-[A-Z]+-[A-Z]+-[A-Z]+)>`.
- The token includes the prefix so it's self-describing: a corrupted registration list still leaves the token parseable.

The tokenizer is added to:

- Write: `WriteKit/Sources/WriteKit/Markdown/MarkdownTokenizer.swift` (existing file).
- Do: `DoKit/Sources/DoKit/Markdown/` (new — Do's notes field currently renders plain text; a minimal tokenizer is introduced here just for references, with a path to fuller markdown later).

In Write, autolinks already lex as a single token; we add a sub-case that recognizes the reference shape and emits a `.reference(kind, wordlyID)` token instead of a generic `.autolink`.

## 4. Backend additions

### 4.1 New `/titles` endpoint per backend

**Write sync** (`~/github/write/sync/`):
```
GET /write/sync/titles?since=<seq>&limit=200
Authorization: Bearer <WRITE_API_KEY>

→ {
  cursor: 4271,
  items: [
    {wordly_id: "W-COPPER-DRIFTING-LANTERN", title: "Project notes", path: "Project notes.md", mtime: 1715800000000, deleted: false},
    …
  ]
}
```

**Sleeper tasks** (`~/chat/tasks/`):
```
GET /tasks/titles?since=<seq>&limit=200
Authorization: Bearer <TASKS_API_KEY>

→ {
  cursor: 1893,
  items: [
    {wordly_id: "DO-RABBIT-WHISPERING-MAUVE", title: "Migrate auth middleware", status: "todo", mtime: 1715800000000, deleted: false},
    …
  ]
}
```

Both endpoints:

- Bearer auth, matching the rest of their respective services.
- Cursor-based, monotonic `seq` (the same `server_seq` write-sync already uses; equivalent on the tasks side).
- Soft-delete-aware: deleted rows are returned with `deleted: true` once, so mirrors can prune.
- Cheap: title + a few small fields per row, no body content. Target ~150 bytes per item, so a full sync of 1,000 items is ~150 KB.
- Stable ordering: by `seq` ascending. The caller paginates by passing `cursor` until `items` is empty.

### 4.2 Schema migrations

**Write sync** — `sync/migrations/00X_add_wordly_id.sql`:

```sql
ALTER TABLE documents ADD COLUMN wordly_id TEXT;
CREATE UNIQUE INDEX idx_documents_wordly_id ON documents(wordly_id) WHERE wordly_id IS NOT NULL;
-- Backfill is not done in SQL. On first /pull after migration, the server emits
-- rows with wordly_id = NULL. The client generates a wordly_id for any such row
-- it owns (sees in its local DocumentStore) and writes it back to the doc's
-- YAML frontmatter, which flows through normal /push.
```

**Sleeper tasks** — analogous migration adding `wordly_id TEXT` to the `tasks` table with a unique partial index. Backfill is server-side and one-shot at migration time: a small node script iterates all tasks, generates a `wordly_id` via the same algorithm as the Swift package (we ship one TypeScript helper for this case — see §6.2), and updates the row. After the script runs, the column is NOT NULL going forward, enforced at the app layer (no DB-level NOT NULL since SQLite doesn't allow adding a NOT NULL column without a default).

The asymmetry (Write backfills client-side via frontmatter; tasks backfill server-side) is deliberate:

- Write docs are authoritative on disk in markdown files — `wordly_id` belongs in the file, so the client writing it via `/push` is the right model.
- Tasks are authoritative in the tasks SQLite — `wordly_id` is a row attribute, so the server backfills.

### 4.3 ID storage

- **Write doc:** `wordly_id` lives in YAML frontmatter as `id: W-…`. `DocumentStore.ensureFrontmatter` runs on open alongside the existing `ensureH1`: if no frontmatter, prepend a minimal block (`---\nid: W-…\n---\n\n`); if frontmatter exists but lacks `id`, insert it. The sync server's `documents.wordly_id` column is a denormalized cache populated from the frontmatter on `/push`. The file is the source of truth.
- **Do task:** `wordly_id` is a column on the `tasks` table. No file-level analogue (tasks don't live as files). The DoTask Swift model gains a `wordlyID: String` property.

### 4.4 API changes for existing endpoints

- `POST /write/sync/push` — accepts (and requires, for new docs) `wordly_id` parsed out of the doc body's frontmatter. If the client sends a doc without frontmatter `id` and the server has none on file for that path, the server rejects with `400 missing wordly_id`. Existing docs pushed by an updated client get a `wordly_id` filled in on the first edit after the migration (the client generates it, writes it into frontmatter, pushes); pushes from non-updated clients for docs that have never received a `wordly_id` are rejected. Mac and iOS Write are updated together, so this gap is the brief window between migration ship and the next time each doc is touched.
- `GET /write/sync/pull` — returns `wordly_id` in every document row.
- Tasks `POST /tasks` / `PATCH /tasks/:id` — accept `wordly_id` on create; server generates if absent (so the existing sleeper-tasks CLI keeps working without changes). `GET /tasks/:id` returns `wordly_id` in the payload.

## 5. Client side

### 5.1 TitleMirror

A new actor in `WordlyRefs`:

```swift
public actor TitleMirror {
    public init(kinds: [any ReferenceKind.Type], storage: URL)

    public func refresh() async throws        // pulls /titles for each kind
    public func search(query: String, kind: any ReferenceKind.Type, limit: Int) -> [Hit]
    public func resolve<K: ReferenceKind>(_ wordlyID: String, as: K.Type) -> K.Item?
}
```

- Storage: a single SQLite file (`titles.sqlite`) with one table per registered kind (`write_titles`, `do_titles`, …). Schema: `(wordly_id TEXT PRIMARY KEY, title TEXT NOT NULL, status TEXT, mtime INTEGER NOT NULL, deleted INTEGER NOT NULL DEFAULT 0, raw_json BLOB)`.
- Search: case-insensitive, title prefix-first then substring, ordered by recency (`mtime DESC`). Returns at most `limit` rows (palette uses 8). Local SQLite scan over a few thousand titles is sub-millisecond.
- Refresh cadence:
  - Pull on app launch (per kind).
  - Pull every 5 minutes on a timer (per kind).
  - Pull immediately after any SSE event from a backend (sleeper-tasks already emits SSE; Write does not — for Write, the existing pull-on-activity loop in `SyncEngine` is good enough).
- Cursor stored in `UserDefaults` per kind: `titles.cursor.<prefix>`.

Mirror storage path:
- Write Mac: `~/Library/Application Support/Write/titles.sqlite`
- Write iOS: app sandbox `Application Support/titles.sqlite`
- Do Mac: `~/Library/Application Support/Do/titles.sqlite`
- Do iOS: app sandbox `Application Support/titles.sqlite`

### 5.2 Slash palette

Triggered when, inside the editor, the user types `/` at a word boundary (start of line, or after whitespace) followed by a registered trigger word and a space. So `/do `, `/write ` — but not `path/do/thing` or `12/write/something` because the `/` isn't at a word boundary.

UI: a popover anchored to the caret. Almanac-themed: warm-paper background, hairline border, two-pixel rust accent on the focused row. Each row shows the item's title (and status pill for `DoKind`).

Behavior:

- As the user types, the query is forwarded to `TitleMirror.search(query:kind:limit: 8)`.
- ↑/↓ move focus, Enter/Tab insert the focused row's reference token at the cursor and dismiss the palette.
- Esc dismisses without insertion (the user's typed `/do query` stays in the text — no auto-delete, since they may have meant it).
- Mouse/touch: clicking a row inserts.
- The palette intercepts the typed `/do ` / `/write ` prefix only after a match selection — until then, the typed characters stay in the buffer. On selection, the editor replaces the range `[start-of-trigger, current-cursor]` with the reference token.

Lives in `WordlyRefs/SlashPalette.swift`. The host app provides:

- A `TextInputBridge` protocol (cursor position, surrounding text, text replacement) — implemented by `MarkdownTextView` in Write and by a small SwiftUI `TextEditor` wrapper in Do.
- The `TitleMirror` instance.

### 5.3 Chip rendering

For each `.reference(kind, wordlyID)` token in the tokenized markdown, the editor's layout substitutes a chip view:

- **WriteKind chip:** `<title>` in the body font, slightly recessed background tint (warm paper at 50% opacity over the page), 1px Almanac-rust border, 4px corner radius. Tap target inflates to full line height.
- **DoKind chip:** `<title>` + a small status glyph (●) colored per `TaskStatus`:
  - `todo` → Almanac rust
  - `draft` → muted ink
  - `recurring` → indigo
  - `done` → green-grey, with strikethrough on the title text
- **Unknown chip** (registered kind but `wordlyID` not in mirror, or kind not registered at all): `⚠ <prefix>:<wordlyID>` in muted ink with a dashed border. Tap surfaces a small popover ("This reference points to an item that isn't synced yet, or no longer exists.").
- **Deleted chip** (mirror has the row with `deleted: true`): `⚠ <title>` (last-known) in muted ink, strikethrough.

Chip view in `WordlyRefs/ChipView.swift`. Both Write's AppKit `MarkdownTextView` and Do's SwiftUI editor render the chip — Write via `NSTextAttachment` with a SwiftUI host; Do via inline `Text` interpolation since Do doesn't have a custom text view yet.

### 5.4 Tap behavior

Resolved at chip-tap time:

- **Same-kind tap inside the host app** (Write chip tapped in Write, Do chip in Do): in-app navigation. Write opens the document into the editor (selecting it in the sidebar/list); Do navigates to the task detail view.
- **Cross-kind tap**: open the URL `<scheme>://<kind>/<wordly_id>`. Concretely:
  - `write://doc/W-COPPER-DRIFTING-LANTERN` opens Write to that document
  - `do://task/DO-RABBIT-DANCING-MAUVE` opens Do to that task

URL scheme registration in each Info.plist (`CFBundleURLTypes`). Each app implements a single route in its `App.onOpenURL { url in … }` handler that parses `scheme://<kind>/<wordly_id>`, looks up the item locally, and navigates.

If the URL's `wordly_id` is unknown to the target app (e.g., Write hasn't yet synced a newly-created note), the target app refreshes its primary sync and then retries the lookup once. If still missing, it shows a small toast: "Couldn't find <prefix>:<wordly_id> yet — try again in a moment."

## 6. Conventions for future apps

Documented in `~/github/wordly-id/README.md`. To add a new kind (e.g., a Links app):

1. Pick a unique uppercase prefix (1–4 letters): `L`.
2. Add a `wordly_id TEXT` column to the kind's authoritative table; backfill once.
3. Implement `GET <base>/titles?since=<seq>&limit=200` returning `{cursor, items: [{wordly_id, title, mtime, deleted, …kind-specific}]}`.
4. Register `link://` URL scheme in the app's Info.plist; route `link://link/<wordly_id>` to the item.
5. In Swift, implement `LinkKind: ReferenceKind` and pass it to `WordlyRefs.registerKinds(…)` at startup.
6. (Optional) Add a TS helper if the backend ever needs to generate IDs (see §6.2).

### 6.1 Wordlist as cross-language invariant

The wordlist is the implicit contract that makes IDs reproducible. v1 ships it only in Swift; if a future backend ever needs to generate IDs server-side, we ship the same `Wordlist.txt` file as part of a tiny `wordly-id-js` npm package alongside the Swift package. The packages share the wordlist file by symlink or duplication-with-CI-check. v1 doesn't need this — only the tasks backfill script reads the wordlist, and we write that script in Node with the wordlist file copied in once.

### 6.2 Why no JS twin in v1

Client-side generation is the rule. The one server-side gen point is the **tasks backfill** at migration time, which is a one-shot script. We hand-author that script reading the same `Wordlist.txt` (committed once to `~/chat/tasks/scripts/wordlist.txt`, with a brief comment pointing at the canonical copy in `~/github/wordly-id/Sources/WordlyID/Wordlist.txt`). After the backfill runs once, no JS path generates IDs again — all new tasks are client-pushed with a client-generated ID.

If we later need ongoing server-side gen (e.g., bot-created items), that's when the `wordly-id-js` package becomes worth building. Not before.

## 8. Data flow (happy path)

User is writing in Write. Types `/do migr`:

```
MarkdownTextView captures keystrokes
  │
  └─ SlashPaletteController detects `/do ` word-boundary trigger
      │
      └─ TitleMirror.search(query: "migr", kind: DoKind, limit: 8)
          │  (local SQLite scan, ~0.1 ms)
          │
          └─ [Hit{wordlyID: "DO-RABBIT-WHISPERING-MAUVE", title: "Migrate auth middleware", status: "todo"}, …]
              │
              └─ Palette renders, user presses Enter
                  │
                  └─ Bridge replaces range "/do migr" → "<do:DO-RABBIT-WHISPERING-MAUVE>"
                      │
                      └─ Doc text now contains the autolink; debounced /push fires
```

Editor render pass (every text change):

```
MarkdownTokenizer sees <do:DO-RABBIT-WHISPERING-MAUVE>
  │
  └─ emits .reference(kind: DoKind, wordlyID: "DO-RABBIT-WHISPERING-MAUVE")
      │
      └─ Chip layout pass calls TitleMirror.resolve(_, as: DoKind)
          │
          └─ {title: "Migrate auth middleware", statusGlyph: "●", deleted: false}
              │
              └─ ChipView rendered inline with rust dot + title
```

User taps the chip:

```
ChipView onTap
  │
  └─ ReferenceRouter.handle(.reference(DoKind, "DO-RABBIT-WHISPERING-MAUVE"), hostKind: WriteKind)
      │
      └─ host ≠ target → open URL `do://task/DO-RABBIT-WHISPERING-MAUVE`
          │
          └─ Do app launches / foregrounds
              │
              └─ App.onOpenURL routes to TaskDetailView(wordlyID: …)
```

## 9. Edge cases

- **Doc has frontmatter `id` that doesn't validate as a `WordlyID`:** treat as if absent, regenerate, overwrite. Log once.
- **Two clients race to assign a `wordly_id` to the same legacy doc:** server's unique index rejects the second push as `409`; client retries with a fresh ID. Surface as `stale` to the existing client loop.
- **Reference points at a kind not registered by this app:** chip renders as "unknown kind". Tapping does nothing. (Open question: do we deep-link to a hypothetical app even if it's not installed? v1: no — too speculative.)
- **Reference target is renamed:** because the reference uses `wordly_id`, the chip's title updates on next mirror refresh. No on-disk text mutation.
- **Reference target is deleted:** mirror marks `deleted: true`. Chip renders strikethrough with last-known title. Tap surfaces a "deleted" popover; in Write the popover offers "Restore from history" since write-sync has `document_revisions`. Do offers no restore in v1.
- **iOS app cold launch via deep link:** the open-url handler waits for `TitleMirror` to do at least one pull before lookup; up to 2 s with a small loading state. After that, the toast fallback.
- **User types `/do ` then keeps typing without selecting:** the palette stays open. If the user types past 60 characters with no match, the palette dismisses silently and the typed text remains.
- **Mirror DB corruption:** delete and recreate on launch. Next sync rebuilds everything; cost is one full pull (~150 KB per kind at 1,000 items).
- **`wordly_id` appears in source as plain text (e.g., the user pastes a token without `<…>` brackets):** not recognized as a reference. Treat as literal text. The token shape is intentionally not a magnet for unintended matches.

## 10. Testing

- **`WordlyID`:** unit tests for `generate` (format, prefix passthrough, retry-on-collision behavior with mock `isUnique`), `parse`, `validate`. Round-trip property test: `validate(generate(prefix:))` holds for all prefixes 1–4 chars.
- **`WordlyRefs`:** unit tests for the tokenizer (recognizes both schemes, ignores look-alikes, handles adjacent tokens); for `TitleMirror` against an in-memory SQLite (refresh cursor advances, deletes prune, search ranking).
- **write-sync:** new `node:test` cases for `/titles` (cursor pagination, soft-delete propagation), and `/push` rejection on missing `wordly_id`.
- **sleeper-tasks:** new `node:test` cases for `/titles` and backfill idempotency.
- **End-to-end:** a small two-client integration test (same harness as the existing write-sync integration tests) that creates a note on client A, references it from a task on client B, and verifies chip resolution.

## 11. Migration / rollout

One ordered sequence:

1. Ship `wordly-id` v0.1 (Swift package only).
2. Ship `~/chat/tasks/` migration + backfill + `/titles` endpoint. Tasks API gains `wordly_id` but nothing consumes it yet.
3. Ship `~/github/write/sync/` migration + `/titles` endpoint.
4. Ship Write + Do client changes together (slash palette, chip rendering, URL schemes, TitleMirror). The two apps depend on each other's backends but not on each other's code.
5. Backfill validation: a one-shot script run from `~` that pulls `/titles` from both backends and asserts every row has a valid `wordly_id`.

Rollback: each step is independently revertible — the columns are nullable until step 5 confirms backfill; the clients tolerate missing `wordly_id` as "unknown chip"; the `/titles` endpoints are additive.

## 12. Open questions

- **Wordlist sourcing.** Start from EFF Long, or hand-curate from scratch? Curated is more on-brand but is real work. Recommendation: start from EFF Long, filter aggressively, hand-polish the result.
- **Frontmatter for non-Write kinds.** If a future kind is also file-based (notes app, journal app), the same frontmatter approach works. Worth noting in the README.
- **Reverse links.** Not in v1, but the data is there in the doc bodies. A future "what links here" panel could be built by adding a server-side index that scans doc bodies for the autolink regex on every `/push`.
- **Status badge for `recurring`.** Do uses `recurring` as a status; the chip color is indigo by current convention. Confirm during the visual pass.

## 13. Design ancestry

- Build on top of the Almanac visual identity that Write and Do already share.
- The slash-palette UX is the Notion/Linear idiom; the autolink syntax borrows from CommonMark's permissive autolink shape; the wordly-id format is a riff on the Diceware passphrase tradition crossed with what.three.words.
